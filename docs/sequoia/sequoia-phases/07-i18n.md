# P7 · i18n Audit — Sequoia CLI

**Audited**: 2026-05-12
**Agent**: P7 sequoia-i18n
**Project**: Sequoia CLI v0.1.0 — Go CLI installer for AI-assistant code audit framework
**Type**: CLI Tool · Developer tool
**Audience**: Developers installing Sequoia into AI assistants (Claude Code, OpenCode, etc.)

---

## Executive Summary

Sequoia CLI has early i18n awareness — a Configuration screen with a language selector (English/Spanish) and a `Language` field threaded through the pipeline. However, the language selection is **completely disconnected from rendering**: every adapter ignores `opts.Language` (with `_ = opts.Language`), all TUI strings are hardcoded English, and there is no translation infrastructure whatsoever.

For a developer CLI tool targeted at an international audience of AI tool users, the lack of i18n is a **medium-priority gap** rather than a critical blocker. The configuration screen and `InstallOpts.Language` field represent good architectural foresight; they just need to be wired up.

**Overall Assessment**: 🟠 RIESGO — Architecture is i18n-aware but unimplemented; language selector is cosmetic.

## Findings

### [P7-001] · Language Selector Exists But Is Not Wired to Any Translation  [🟠 RIESGO]
**State**: Confirmed
**Evidence**:
- `internal/tui/screens/configuration.go:14-19` — Language selector with EN/ES options
- `internal/tui/screens/configuration.go:171` — Language stored in `TUIConfig.Language`
- `internal/app/update.go:370` — Language passed to `pipeline.RunInstall(ctx, tools, ch, config.Language)`
- `adapters/claude/adapter.go:126` — `_ = opts.Language` (ignored)
- `adapters/opencode/adapter.go:126` — `_ = opts.Language` (ignored)
- Every other adapter ignores `opts.Language` identically

**Problem**: The TUI lets users select English or Spanish from a Configuration screen, and the choice is threaded through the pipeline to `InstallOpts.Language`. However, every adapter discards it with `_ = opts.Language`. No translation files exist, no string lookups depend on the language setting, and the TUI itself renders all strings in hardcoded English regardless of the user's selection.

**Real Impact**: Non-English-speaking developers see an "Español" option that does nothing. The UI remains entirely in English. User expectations set by the language selector are violated — a trust and UX issue.

**Minimal High-Leverage Recommendation**: Either wire the language selector to actual translation files (recommended for v0.2+) or hide it until implemented (immediate fix). The infrastructure is already in place — `opts.Language` just needs consumers.

**Dependencies/Blockers**: Requires translation of all TUI strings and templates before wiring.

**Implementation Risk**: Medium — 51 user-facing strings need translation; templates (skill.md, agents docs) need per-language variants; no existing message catalog.

**Acceptance Criteria**:
- [ ] Selecting "Español" on the Configuration screen changes TUI strings to Spanish
- [ ] Template rendering selects language-appropriate files (e.g., `templates/es/skill.md.tmpl`)
- [ ] `adapter.Install(opts)` uses `opts.Language` to select language variants
- [ ] At least 2 adapters (Claude, OpenCode) support language selection

**Problem**: Every user-facing string in the TUI is a Go string literal. There is no message catalog, no i18n key constants, and no mechanism to extract strings for translation. A total of approximately 51 distinct user-facing strings are scattered across 8 screen files and the app model, all in English.

**Real Impact**: Adding a new language requires manually editing every screen file to add conditional logic (`if lang == "es" { ... }`), which is error-prone and unscalable. Even the existing en/es toggle cannot work without this infrastructure.

- [ ] `T("welcome.menu.install", lang)` returns the correct language string
- [ ] English strings produce identical output to current behavior (regression)
- [ ] Spanish strings are provided with native-quality translations

### [P7-003] · Embedded Agent Templates Are Mixed-Language With No Variant Selection  [🟡 ATENCIÓN]
**State**: Confirmed
**Evidence**:
- `adapters/claude/templates/skill.md.tmpl:16` — `"Eres el orquestador de Sequoia. Tu función es coordinar la auditoría técnica..."` (Spanish)
- `adapters/claude/templates/skill.md.tmpl` — Entire 176-line orchestrator prompt in Spanish
- `adapters/opencode/templates/skill.md.tmpl:16` — Same Spanish content as Claude's
- `adapters/claude/templates/claude-md-section.md.tmpl:1-29` — 100% English
- `adapters/opencode/templates/agents-md-section.md.tmpl:1-41` — 100% English
- `adapters/common/commands.go:6-12` — Command filenames: English-only markdown files
- Every adapter's `Install()` accepts `opts.Language` but ignores it when rendering templates

**Problem**: The installed agent prompts (SKILL.md) are written entirely in Spanish with instructions like "Eres el orquestador...", while the system prompt sections (CLAUDE.md, AGENTS.md) are in English. There is no mechanism to select a language-specific template variant. A French or Japanese developer installing Sequoia will get Spanish agent instructions embedded in their English system prompts — a jarring mixed-language experience.

**Real Impact**: Non-English, non-Spanish developers get a confusing mix of English and Spanish in their installed Sequoia content. The orchestrator skill (the primary interaction point) is only available in Spanish, limiting the tool to Spanish+English bilingual developers and excluding the global developer audience.

**Minimal High-Leverage Recommendation**: Keep the current Spanish skill as the `es` variant, create an English translation as the default, and add template directory structure like `templates/{lang}/` with `common.RenderTemplate` selecting the language-appropriate file based on `opts.Language`.

**Dependencies/Blockers**: Requires translation of the 176-line skill.md.tmpl and all command markdown files to English.

**Implementation Risk**: Low — additive; add new template directories without modifying existing behavior for Spanish users.

**Acceptance Criteria**:
- [ ] English skill.md.tmpl created with native-quality translation
- [ ] Template rendering selects `templates/{lang}/skill.md.tmpl` based on `opts.Language`
- [ ] All 5 command files (`sequoia-*.md`) have English versions as default
- [ ] Spanish templates preserved in `templates/es/` subdirectories
- [ ] Default language is English when no language is specified

### [P7-004] · No i18n Library or Translation Infrastructure  [🟡 ATENCIÓN]
**State**: Confirmed
**Evidence**:
- `go.mod:1-37` — No i18n libraries present (no `gotext`, `go-i18n`, `nicksnyder/go-i18n`, `x/text/message`)
- `grep` across entire codebase for `i18n`, `locale`, `translation`, `gotext`, `message.*catalog` — Only matches found are documentation references in agent prompt templates and test data, not actual code
- No `.json`, `.toml`, `.yaml` message files anywhere in the project
- No plural rules engine or locale-aware formatting utilities

**Problem**: The project has zero i18n infrastructure. There is no library for message lookups, no format for translation files, no mechanism for pluralization, and no locale-aware date/number/currency formatting. The `Language` type in `model/types.go` is unused by any rendering code.

**Real Impact**: Even if all strings were externalized (P7-002), there is no runtime mechanism to perform the lookup. Adding i18n would require building everything from scratch.

**Minimal High-Leverage Recommendation**: Add `github.com/nicksnyder/go-i18n/v2` (lightweight, 0 runtime deps beyond `golang.org/x/text`) to `go.mod`. Define message files in JSON or TOML format under `internal/i18n/translations/`. This library handles pluralization, interpolation, and file-based message loading.

**Dependencies/Blockers**: None — greenfield addition.

**Implementation Risk**: Low — add a dependency and create translation files; no existing code paths modified until P7-002 is tackled.

**Acceptance Criteria**:
- [ ] `go-i18n/v2` added to `go.mod` and `go.sum`
- [ ] `internal/i18n/translations/en.json` with all 51+ TUI strings
- [ ] `internal/i18n/translations/es.json` with all 51+ TUI strings
- [ ] `internal/i18n/bundle.go` initializes the `i18n.Bundle` with both languages
- [ ] `T(ctx, key)` function returns translated string for language in context

### [P7-005] · CLI Help Text (Cobra) Is English-Only With No Localization Path  [🟡 ATENCIÓN]
**State**: Confirmed
**Evidence**:
- `cmd/sequoia/main.go:64-76` — Root command `Use`, `Short`, `Long` — all English
- `cmd/sequoia/main.go:103-116` — Install command — all English
- `cmd/sequoia/main.go:136-142` — Status command — all English
- `cmd/sequoia/main.go:160-173` — Uninstall command — all English
- `cmd/sequoia/main.go:187-191` — Version command — all English
- `cmd/sequoia/main.go:57` — Error output: `fmt.Fprintf(os.Stderr, "error: %v\n", err)` — English
- `cmd/sequoia/main.go:205-346` — All headless output messages are English string literals (`"No supported AI tools detected"`, `"Installing Sequoia for %s ..."`, etc.)

**Problem**: All Cobra command descriptions, flag help text, and headless-mode output are hardcoded English string literals. Cobra has no built-in i18n mechanism. Users running `sequoia --help`, `sequoia install --help`, or headless commands always get English output regardless of locale.

**Real Impact**: Non-English developers using the CLI in headless mode (CI/CD, scripts) see all output in English. Since this is a developer tool, this is lower priority than the TUI, but still excludes non-English-speaking developers from self-service CLI usage.

**Minimal High-Leverage Recommendation**: Use the same `internal/i18n` catalog from P7-004 to feed Cobra's `Use`/`Short`/`Long` fields. For headless output, replace `fmt.Fprintf` calls with `fmt.Fprintf(out, i18n.T("cli.install.progress", lang), a.Name())`.

**Dependencies/Blockers**: Depends on P7-004 (i18n infrastructure) and P7-001 (language detection).

**Implementation Risk**: Low — Cobra commands are created at startup; language can be passed to `newRootCmd(lang)` or read from environment/settings.

**Acceptance Criteria**:
- [ ] `--help` output respects the configured language
- [ ] Headless install/status/uninstall output uses translated strings
- [ ] Fallback to English when no language is configured

### [P7-006] · Error Messages Are English-Only With No Structured Error Codes  [🟡 ATENCIÓN]
**State**: Confirmed
**Evidence**:
- `cmd/sequoia/main.go:57` — `"error: %v\n"` — bare English wrapper
- `cmd/sequoia/main.go:213` — `"unknown adapter %q — use 'sequoia status'..."` — English
- `cmd/sequoia/main.go:215` — `"No supported AI tools detected on this machine."` — English
- `cmd/sequoia/main.go:216` — `"Currently supported: Claude Code, OpenCode..."` — English
- `cmd/sequoia/main.go:302` — `"unknown adapter %q — use 'sequoia status'..."` — English
- `cmd/sequoia/main.go:304` — `"No adapters to uninstall from."` — English
- `cmd/sequoia/main.go:311` — `"stdin is not a terminal; use --yes to skip confirmation"` — English
- `internal/app/update.go:83` — `"Select at least one tool to continue"` — English
- `internal/app/update.go:223` — `"Select at least one installed tool to continue"` — English
- `adapters/claude/adapter.go:130-204` — All `fmt.Errorf("install: ...")` messages — English
- `adapters/opencode/adapter.go:130-207` — All `fmt.Errorf("install: ...")` messages — English
- `adapters/interface.go:7` — `ErrUnknownAdapter = errors.New("unknown adapter")` — English sentinel value

**Problem**: Error messages scattered across 10+ files are all English string literals. There are no error codes, no error categories, and no mechanism to translate error messages. Even the sentinel error `ErrUnknownAdapter` is an English string that can't be translated without changing its identity (used in `errors.Is` comparisons).

**Real Impact**: Non-English-speaking developers troubleshooting installation failures see English error messages they may not understand. Since installation is the primary use case, this directly impacts the core user journey.

**Minimal High-Leverage Recommendation**: Introduce structured error types with machine-readable codes (e.g., `"ADAPTER_NOT_FOUND"`) and human-readable messages from the i18n catalog. Sentinels like `ErrUnknownAdapter` should use `errors.New` with a code-based identity while the display message comes from the catalog.

**Dependencies/Blockers**: Depends on P7-004 (i18n infrastructure).

**Implementation Risk**: Medium — error handling changes touch many files; sentinel error changes may break `errors.Is` checks in tests.

**Acceptance Criteria**:
- [ ] Error types include a machine-readable code field
- [ ] Error display uses i18n catalog for human-readable messages
- [ ] Sentinel errors (`ErrUnknownAdapter`) preserve `errors.Is` compatibility
- [ ] Installation failure screen shows translated error messages

### [P7-007] · No RTL Layout or Unicode Bidirectional Text Support  [🟢 INFO]
**State**: Confirmed
**Evidence**:
- `go.mod:1-37` — Bubbletea v1.3.10 + Lipgloss v1.1.0 — neither provides native RTL support
- `internal/tui/screens/install-progress.go:48` — `spinnerFrames = []string{"⠋", "⠙", "⠹", "⠸", "⠼", "⠴", "⠦", "⠧", "⠇", "⠏"}` — Braille spinners (Unicode, LTR-rendered)
- `internal/tui/screens/complete.go:22` — `"✅  Installation Complete!"` — Emoji + text, LTR layout
- All screens use left-to-right indentation (`"  "`, `"    "`) and left-aligned text
- No `unicode.Bidi` usage or RTL-aware rendering logic anywhere

**Problem**: The Bubbletea/Lipgloss stack renders text left-to-right by default. There is no Bidirectional Algorithm implementation, no RTL layout mode, and no mechanism to mirror the UI (right-align text, flip arrow indicators, reposition emoji). Terminal emulators handle basic bidirectional text, but composed layouts (cursor indicators, checkboxes, progress bars) will not render correctly in RTL.

**Real Impact**: Minimal. This is a developer CLI tool used in terminal emulators that support Unicode. Arabic, Hebrew, and Persian developers can use the English UI. RTL support for a developer tool is a very low priority — most developer tools (git, npm, go, docker) don't support RTL either.

**Minimal High-Leverage Recommendation**: No action needed for v0.x. If i18n is pursued for v1.0+, consider RTL layout only if there is demonstrated demand from RTL-language developer communities. Document the limitation.

**Dependencies/Blockers**: Would require RTL support in Bubbletea/Lipgloss (upstream dependency) or a custom rendering layer.

**Implementation Risk**: High — requires upstream changes or a parallel rendering pipeline; very high effort-to-impact ratio.

**Acceptance Criteria**:
- [ ] RTL limitation documented in project README or CONTRIBUTING.md
- [ ] No action required for v0.x
