# Mandatory Phase Template

Structure that EVERY phase document generated by `sequoia-reporter` must follow. No exceptions.

## Template

```markdown
# Phase [N] — [Name]

**Agent**: sequoia-[name]
**Project**: [name]
**Audit date**: [date]
**Detected stack**: [from project map]
**Mode**: Full | Quick | Review

---

## 1. Phase objective

What domain this phase covers and what it seeks to evaluate. 2-3 lines.

## 2. Inspection scope

### Files and directories reviewed
- `{path/}` — [reason for inclusion]
- `{path/}` — [reason for inclusion]

### What was left out of scope
- `{path/}` — [reason for exclusion: generated, vendor, out of context]
- `{path/}` — [reason for exclusion]

## 3. Verified current state

### What internal documentation says (if it exists)
[Summary of relevant docs. If no docs: "No relevant internal documentation was found."]

### What was confirmed in code
[Facts verified against real code.]

### What was outdated, ambiguous, or not verifiable
- [If there are docs that contradict the code, list here]
- [If there are aspects that couldn't be verified, declare with `[NOT VERIFIABLE]` or `[REQUIRES EXTERNAL ACCESS]`]

## 4. Consolidated findings

Ordered by severity (🔴 → 🟠 → 🟡). No duplicates. No filler.

### [{AGENT}-{NNN}] · [Finding title]  [🔴 CRITICAL | 🟠 RISK | 🟡 ATTENTION]

**Status**: Confirmed | Partial | Not verifiable | Outdated

**Evidence**:
- `path/to/real/file.ext:line` — description

**Problem**:
[Concrete technical description]

**Real impact**:
[What happens in production]

**Minimum high-leverage recommendation**:
[A single maximum-impact action]

**Dependencies / blockers**:
[If applicable]

**Implementation risk**: Low | Medium | High
[Reason]

**Acceptance criteria**:
- [ ] [verifiable condition]

---

*(Repeat for each finding)*

## 5. High-leverage missing items

Only technically justified improvements. With expected impact. No wishlist.

| Improvement | Expected impact | Estimated effort |
|--------|-----------------|-------------------|
| [description] | [what changes] | [low/medium/high] |

## 6. Task plan

### [TASK-{N}.{NNN}] · [Task title]

- **Context**: [why it's needed, 2-3 lines]
- **Candidate files/modules**: `path/to/file`
- **Impact on current app**: [what improves]
- **Prior dependency**: [TASK-XX.XXX if applicable, or "none"]
- **Breakage risk**: Low | Medium | High — [reason]
- **Acceptance criteria**:
  - [ ] [verifiable condition]
- **Priority**: 🔴 Blocking | 🟠 High leverage | 🟡 Backlog

---

*(Repeat for each task)*

## 7. Recommended implementation order

Sequence that minimizes risk and maximizes impact:

```
1. [TASK-XX.001] → root cause, unblocks other tasks
2. [TASK-XX.003] → dependent on 001, quick win
3. [TASK-XX.002] → independent, can be done in parallel
4. [TASK-XX.004] → requires 001 and 003 to be complete
```

Order justification: [why this sequence]

## 8. Phase risks and blockers

| Risk/Blocker | Impact | Mitigation |
|---------------|---------|------------|
| [description] | [high/medium/low] | [suggested action] |

## 9. Phase closure checklist

Verifiable list: what must be true when this phase is "done."

- [ ] All 🔴 findings have assigned tasks
- [ ] All tasks have acceptance criteria
- [ ] Dependencies between tasks are declared
- [ ] The implementation order is justified
- [ ] Unverifiable findings are marked as such
- [ ] No generic recommendations without specific evidence
- [ ] Files cited in evidence exist and are correct
```

## Template rules

1. **No empty sections**. If a section does not apply, write "Not applicable: [reason]" instead of leaving it blank.
2. **Every finding uses the standard format** from `references/finding-format.md`.
3. **Each task is self-contained**: does not require reading other sections to be understood.
4. **Sections 1-3 are context**, 4-6 are actionable, 7-9 are closure.
5. **The closure checklist** must be verifiable without ambiguity.
